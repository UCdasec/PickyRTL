This code example is from the Hack@DAC 2018 Phase 2 Buggy SoC. It is from the md5 module.
The 'data_o' output signal is the reuslt output for the module. Within the 'always @(posedge clk or negedge reset)' 
procedural block 'data_o' is assigned the value of 'next_data_o' which may hold intermediate results
of the md5 algorithm. This is an example of the CWE-1431 vulnerability as these intermediate values
should not be assigned to the module output unless valid. I have implemented a fix to show a correct
design pattern and follows the coding style of the module.
