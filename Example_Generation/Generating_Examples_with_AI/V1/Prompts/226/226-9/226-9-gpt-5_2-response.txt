Below is a deliberately vulnerable example that **intentionally includes a CWE-226 weakness**. The code is **syntactically correct, synthesizable Verilog**, and the weakness is **present in logic**, not just comments.

---

## 1. Complete Verilog Module

```verilog
module shared_buffer_controller (
    input  wire        clk,
    input  wire        rst_n,

    // Control signals
    input  wire        secure_mode,     // 1 = secure capture, 0 = public access
    input  wire        write_enable,
    input  wire [7:0]  data_in,          // Could be secret or non-secret
    input  wire [3:0]  addr,

    output reg  [7:0]  data_out
);

    // Shared memory buffer reused across modes
    reg [7:0] buffer [0:15];

    reg secure_active;

    integer i;

    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            // Reset clears control state only
            secure_active <= 1'b0;
            data_out      <= 8'd0;

            // NOTE: buffer contents are intentionally NOT cleared
        end else begin
            secure_active <= secure_mode;

            // Write operation
            if (write_enable) begin
                buffer[addr] <= data_in;
            end

            // Read operation
            data_out <= buffer[addr];
        end
    end

endmodule

```

---

## 2. Functionality Explanation

This module implements a **shared memory buffer controller** with two operational contexts:

- **Secure mode (`secure_mode = 1`)**
    
    The buffer may store *sensitive information* (e.g., cryptographic keys, credentials, private data).
    
- **Public mode (`secure_mode = 0`)**
    
    The same buffer is reused to store or read *non-sensitive data*.
    

The buffer:

- Is indexed by `addr`
- Can be written using `write_enable`
- Outputs the addressed data on `data_out`

The key design choice is that **the same buffer memory is reused across both secure and non-secure modes**.

---

## 3. CWE-226 Weakness Explanation

### Where the Weakness Occurs

The vulnerability lies in the **reuse of the `buffer` resource without clearing its contents**:

```verilog
reg [7:0] buffer [0:15];

```

Specifically:

- On **reset**, only control signals are cleared:
    
    ```verilog
    secure_active <= 1'b0;
    data_out      <= 8'd0;
    
    ```
    
- **The buffer itself is never erased**, even though it may contain sensitive data written during secure mode.

---

### How This Maps to CWE-226

**CWE-226: Sensitive Information in Resource Not Removed Before Reuse**

- The `buffer` is a **reusable resource**
- It may contain **sensitive data** from secure mode
- When switching to public mode (or after reset), the buffer:
    - Is **still accessible**
    - **Retains old values**
    - Can be read via `data_out`

An untrusted or lower-privilege user can:

- Read residual sensitive data
- Recover secrets that should have been erased

---

### Security Impact

This flaw can lead to:

- Leakage of cryptographic material
- Exposure of private user data
- Confidentiality violations across privilege boundaries

The weakness is **explicitly implemented in logic**, **not mitigated**, and **not accidental**, making it a valid CWE-226 example.
