Below is an intentionally **insecure** example that demonstrates **CWE-226: Sensitive Information in Resource Not Removed Before Reuse** in synthesizable Verilog.

---

## 1. Complete Verilog Module

```verilog
module insecure_key_buffer (
    input  wire        clk,
    input  wire        reset_n,      // Active-low reset
    input  wire        secure_mode,   // 1 = load secret, 0 = normal operation
    input  wire [31:0] data_in,
    output reg  [31:0] data_out
);

    // This register is used to store sensitive data (e.g., a secret key)
    // and later reused as a general-purpose buffer.
    reg [31:0] buffer_reg;

    always @(posedge clk) begin
        if (!reset_n) begin
            // INTENTIONAL WEAKNESS:
            // buffer_reg is NOT cleared on reset.
            // data_out is cleared, but the sensitive buffer is retained.
            data_out <= 32'b0;
        end else begin
            if (secure_mode) begin
                // In secure mode, load sensitive information
                buffer_reg <= data_in;     // e.g., cryptographic key
                data_out   <= 32'b0;
            end else begin
                // In normal mode, the same register is reused
                // without clearing its previous sensitive contents.
                data_out   <= buffer_reg;  // Leaks old sensitive data
                buffer_reg <= data_in;     // Reused as a normal buffer
            end
        end
    end

endmodule

```

---

## 2. Functionality of the Module

This module models a simple data buffer with two operational modes:

- **Secure mode (`secure_mode = 1`)**
    - `buffer_reg` stores sensitive input data (for example, a cryptographic key).
    - No output is produced during this mode.
- **Normal mode (`secure_mode = 0`)**
    - The same `buffer_reg` is reused as a general-purpose data buffer.
    - Its contents are driven directly to `data_out`.
    - New non-sensitive data overwrites the register afterward.

The module includes a reset signal (`reset_n`), which resets the output but **does not reset internal storage**.

---

## 3. Where and How the CWE-226 Weakness Is Introduced

### Location of the Weakness

- The weakness is centered on the `buffer_reg` register.
- Specifically:
    - `buffer_reg` is **not cleared on reset**
    - `buffer_reg` is **reused across security boundaries** (secure → normal mode)

### How the Weakness Manifests

- When the module operates in **secure mode**, `buffer_reg` holds sensitive information.
- Upon reset or transition to **normal mode**, that sensitive data:
    - Remains intact in `buffer_reg`
    - Is subsequently exposed via `data_out`

### Why This Is CWE-226

This is a textbook case of **Sensitive Information in Resource Not Removed Before Reuse**:

- A storage resource (`buffer_reg`) previously holding sensitive data
- Is reused without being cleared
- Allowing less-trusted logic to observe residual secret values

### Security Impact

An attacker or untrusted subsystem observing `data_out` in normal mode could recover sensitive information that should have been destroyed, violating confidentiality guarantees.
