## 1. Complete Verilog Module (intentionally vulnerable)

```verilog
module insecure_buffer_reuse (
    input  wire        clk,
    input  wire        rst,          // Active-high synchronous reset
    input  wire        secure_mode,  // 1 = trusted writer, 0 = untrusted reader
    input  wire        write_en,
    input  wire        read_en,
    input  wire [3:0]  addr,
    input  wire [31:0] data_in,
    output reg  [31:0] data_out
);

    // Memory resource reused across security domains
    reg [31:0] buffer [0:15];

    integer i;

    always @(posedge clk) begin
        if (rst) begin
            // Reset logic does NOT clear buffer contents
            data_out <= 32'b0;
        end else begin
            // Secure mode writes sensitive data
            if (secure_mode && write_en) begin
                buffer[addr] <= data_in;
            end

            // Non-secure mode can read from the same buffer
            if (!secure_mode && read_en) begin
                data_out <= buffer[addr];
            end
        end
    end

endmodule

```

---

## 2. Functionality Explanation

This module implements a small 16-word memory buffer shared between two operational modes:

- **Secure mode (`secure_mode = 1`)**
    - Intended for trusted logic
    - Allows writing data (potentially sensitive, e.g., cryptographic material) into the buffer
- **Non-secure mode (`secure_mode = 0`)**
    - Intended for untrusted or lower-privilege logic
    - Allows reading data from the same buffer

A synchronous reset initializes the output register (`data_out`) but leaves the internal memory buffer unchanged.

---

## 3. CWE-226 Weakness Explanation

**Sensitive Information in Resource Not Removed Before Reuse**

### Where the weakness is introduced

```verilog
if (rst) begin
    data_out <= 32'b0;
end

```

The reset logic **does not clear the `buffer` memory**, which previously stored sensitive data written in secure mode.

### How the weakness manifests

- Sensitive data is written into `buffer` while in secure mode
- A reset or mode transition occurs
- The buffer resource is reused in non-secure mode
- Residual sensitive data remains intact and readable
- Untrusted logic can read old values via `read_en`

### Why this is CWE-226

- The memory buffer is a **reused resource**
- It contains **sensitive information**
- The data is **not removed or cleared before reuse**
- This creates a **confidentiality breach** across security boundaries

This is a textbook CWE-226 pattern involving retained register/memory contents across reset or privilege transitions, intentionally left unmitigated.
