Below is a **self-contained, synthesizable Verilog HDL module** that **intentionally includes a CWE-226 weakness** (Sensitive Information in Resource Not Removed Before Reuse). The weakness is present in the logic itself and is **not mitigated**.

---

## 1. Complete Verilog Module

```verilog
module insecure_resource_reuse (
    input  wire        clk,
    input  wire        reset_n,

    // Control signals
    input  wire        load_secret,   // Load sensitive data
    input  wire        switch_to_user, // Switch from privileged to user mode
    input  wire        user_read,      // User attempts to read data

    // Data inputs
    input  wire [31:0] secret_in,      // Sensitive value (e.g., cryptographic key)

    // Data output
    output reg  [31:0] data_out
);

    // Simple mode encoding
    localparam MODE_PRIVILEGED = 1'b0;
    localparam MODE_USER       = 1'b1;

    reg mode;

    // Register that stores sensitive information
    reg [31:0] sensitive_reg;

    always @(posedge clk) begin
        if (!reset_n) begin
            // Reset only sets the mode
            // NOTE: sensitive_reg is NOT cleared here
            mode     <= MODE_PRIVILEGED;
            data_out <= 32'b0;
        end else begin
            // Load sensitive data in privileged mode
            if (mode == MODE_PRIVILEGED && load_secret) begin
                sensitive_reg <= secret_in;
            end

            // Transition to user mode
            if (switch_to_user) begin
                mode <= MODE_USER;
            end

            // User can read data once in user mode
            if (mode == MODE_USER && user_read) begin
                // Reuse of sensitive_reg without clearing
                data_out <= sensitive_reg;
            end
        end
    end

endmodule

```

---

## 2. Module Functionality Explanation

This module models a **simple privileged-to-user mode transition**:

- In **privileged mode**, the module can load sensitive data (e.g., a cryptographic key) into an internal register (`sensitive_reg`).
- A control signal switches the module into **user mode**, representing a lower-privilege context.
- In user mode, an external user can request a read operation.
- The module outputs data from an internal register onto `data_out`.

The design intentionally reuses internal storage across security boundaries.

---

## 3. CWE-226 Weakness Explanation

### Where the Weakness Is Introduced

The weakness is introduced in the handling of the `sensitive_reg` register:

```verilog
reg [31:0] sensitive_reg;

```

- This register stores **sensitive information** when the module is in privileged mode.
- When the module transitions to user mode, **the register is neither cleared nor invalidated**.
- On reset, the register is also **left untouched**, retaining any previously stored value.

### How the Weakness Manifests

```verilog
if (mode == MODE_USER && user_read) begin
    data_out <= sensitive_reg;
end

```

- The same register that once held sensitive data is **reused as a data source** in user mode.
- Because its contents were never cleared, **residual sensitive information becomes observable** by a lower-privilege user.

### Why This Is CWE-226

This directly matches **CWE-226: Sensitive Information in Resource Not Removed Before Reuse**:

- A resource (register) containing sensitive data is **reused across security boundaries**
- The sensitive data is **not removed before reuse**
- This enables **confidentiality violations through data remanence**
