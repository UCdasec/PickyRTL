Below is an intentionally vulnerable, **self-contained and synthesizable Verilog HDL module** that demonstrates **CWE-1233: Security-Sensitive Hardware Controls with Missing Lock Bit Protection**.

---

## 1. Complete Verilog Module

```verilog
module insecure_key_ctrl (
    input  wire        clk,
    input  wire        rst_n,

    // Simple register write interface
    input  wire        wr_en,
    input  wire [3:0]  wr_addr,
    input  wire [31:0] wr_data,

    // Outputs for observation
    output reg  [31:0] secure_key,
    output reg         debug_enable,
    output reg         lock_bit
);

    // Address map
    localparam ADDR_KEY   = 4'h0;
    localparam ADDR_CTRL  = 4'h1;

    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            secure_key   <= 32'h00000000;
            debug_enable <= 1'b0;
            lock_bit     <= 1'b0;
        end else begin
            if (wr_en) begin
                case (wr_addr)

                    // SECURITY-SENSITIVE REGISTER
                    // Intended to be protected by lock_bit
                    ADDR_KEY: begin
                        // VULNERABILITY:
                        // secure_key can be overwritten regardless of lock_bit
                        secure_key <= wr_data;
                    end

                    // Control register
                    ADDR_CTRL: begin
                        // debug_enable is (incorrectly) protected by lock_bit
                        if (!lock_bit) begin
                            debug_enable <= wr_data[1];
                        end

                        // lock_bit itself is writable at any time
                        lock_bit <= wr_data[0];
                    end

                    default: begin
                        // no-op
                    end

                endcase
            end
        end
    end

endmodule

```

---

## 2. Module Functionality Explanation

This module models a simple hardware control block with a register-write interface:

- **`secure_key`**
    
    A 32-bit security-sensitive register representing something critical (e.g., a cryptographic key, root hash, or authorization token).
    
- **`debug_enable`**
    
    A control bit that enables or disables debug functionality.
    
- **`lock_bit`**
    
    A configuration lock intended to prevent modification of security-critical state after system initialization.
    

### Register Map

| Address | Register | Description |
| --- | --- | --- |
| `0x0` | `secure_key` | Security-sensitive key register |
| `0x1` | `lock_bit` / `debug_enable` | Control register |

On reset:

- All registers initialize to zero.
- Software can write registers via `wr_en`, `wr_addr`, and `wr_data`.

---

## 3. CWE-1233 Weakness Explanation

### **Where the Weakness Is Introduced**

The vulnerability exists in the write logic for `secure_key`:

```verilog
ADDR_KEY: begin
    secure_key <= wr_data;
end

```

### **How This Violates CWE-1233**

- A **lock bit (`lock_bit`) exists** and is clearly intended to protect sensitive configuration.
- The lock bit **is actively used** to protect `debug_enable`.
- However, **`secure_key` completely ignores the lock bit**.

This creates a subtle but dangerous situation:

- Reviewers may assume all security-critical registers are protected because:
    - A lock mechanism exists
    - It is used elsewhere in the module
- In reality, an attacker or malicious software can **rewrite the secure key at any time**, even after the system is supposedly “locked.”

### **Why This Is a Security Risk**

Once a system is locked:

- Cryptographic keys should become immutable
- Trust anchors should not change
- Privilege boundaries should remain fixed

Because `secure_key` ignores `lock_bit`, **post-lock tampering is possible**, enabling:

- Key replacement attacks
- Bypass of secure boot or authentication
- Privilege escalation via hardware misconfiguration

This is a textbook example of **CWE-1233: Security-Sensitive Hardware Controls with Missing Lock Bit Protection**, implemented in a **non-obvious, partially protected design**.
