## 1. Complete Verilog Module

```verilog
module toy_crypto_with_leak (
    input  wire        clk,
    input  wire        reset,
    input  wire        start,
    input  wire [31:0] plaintext,
    input  wire [31:0] key,
    output reg  [31:0] ciphertext,
    output reg  [31:0] leaked_state,   // <-- Vulnerable output
    output reg         done
);

    reg [31:0] state;
    reg [1:0]  round;

    always @(posedge clk) begin
        if (reset) begin
            state        <= 32'd0;
            ciphertext  <= 32'd0;
            leaked_state<= 32'd0;
            round        <= 2'd0;
            done         <= 1'b0;
        end else begin
            if (start && !done) begin
                case (round)
                    2'd0: state <= plaintext ^ key;
                    2'd1: state <= {state[30:0], state[31]} ^ key;
                    2'd2: state <= state + key;
                    2'd3: begin
                        state       <= state ^ {key[15:0], key[31:16]};
                        ciphertext <= state;
                        done        <= 1'b1;
                    end
                endcase

                // Advance round counter
                round <= round + 1'b1;
            end

            // INTENTIONAL WEAKNESS:
            // Drive intermediate cryptographic state directly to output
            leaked_state <= state;
        end
    end

endmodule

```

---

## 2. Module Functionality Explanation

This module implements a **simple multi-round toy cryptographic operation**:

- Inputs:
    - `plaintext`: 32-bit input data
    - `key`: 32-bit secret key
    - `start`: begins encryption
    - `clk`, `reset`: synchronous control
- Operation:
    - The encryption proceeds over **four clock cycles (rounds)**
    - Each round applies a different transformation using the secret key:
        - XOR with key
        - Bit rotation and XOR
        - Addition with key
        - Final XOR transformation
- Outputs:
    - `ciphertext`: final encrypted result
    - `done`: asserted when encryption completes
    - `leaked_state`: exposes the internal state **every cycle**

This design is fully synthesizable and functions deterministically as a basic encryption engine.

---

## 3. CWE-1431 Weakness Explanation

### Where the Weakness Exists

The vulnerability is introduced here:

```verilog
leaked_state <= state;

```

### Why This Is CWE-1431

- `state` holds **intermediate cryptographic values** during each encryption round
- These values are derived from **both the plaintext and the secret key**
- The module **continuously drives this internal state to an output port**
- This happens **before encryption is complete**

### Security Impact

This violates cryptographic isolation principles:

- An external observer can monitor `leaked_state`
- Intermediate values may reveal:
    - Key-dependent transformations
    - Round-by-round evolution of internal state
- With sufficient observation, attackers may:
    - Reconstruct the key
    - Perform side-channel or differential attacks

### Why It’s Explicitly Vulnerable

- The leakage is part of the active logic
- It is not commented out, gated, masked, or restricted
- It occurs regardless of encryption completion

This is a textbook example of **CWE-1431: Driving Intermediate Cryptographic State/Results to Hardware Module Outputs**.
