## 1. Complete Verilog Module (Intentionally Vulnerable to CWE-226)

```verilog
// ================================================================
// Multi-Context Key Management Unit (Intentionally Vulnerable)
// Domain: Key management unit
// Weakness: CWE-226 - Sensitive Information in Resource Not Removed Before Reuse
// ================================================================

module key_management_unit #(
    parameter KEY_WIDTH  = 128,
    parameter NUM_SLOTS  = 4,
    parameter USER_WIDTH = 2                 // Supports up to 4 users
)(
    input                       clk,
    input                       rst_n,

    // Allocation interface
    input                       alloc_req,
    input  [USER_WIDTH-1:0]     alloc_user_id,
    input  [KEY_WIDTH-1:0]      alloc_key_data,
    output reg                  alloc_grant,
    output reg  [1:0]           alloc_slot_id,

    // Release interface
    input                       release_req,
    input  [1:0]                release_slot_id,

    // Read interface
    input                       read_req,
    input  [1:0]                read_slot_id,
    output reg [KEY_WIDTH-1:0]  read_key_data,
    output reg                  read_valid,

    // Status
    output reg [NUM_SLOTS-1:0]  slot_in_use
);

    // ============================================================
    // Internal storage resources
    // ============================================================

    // Key storage memory (resource type 1: memory block)
    reg [KEY_WIDTH-1:0] key_storage [0:NUM_SLOTS-1];

    // Ownership tracking (resource type 2: registers)
    reg [USER_WIDTH-1:0] slot_owner [0:NUM_SLOTS-1];

    // FSM state (resource type 3: control register)
    reg [1:0] state;

    localparam IDLE      = 2'd0;
    localparam ALLOCATE  = 2'd1;
    localparam RELEASE   = 2'd2;
    localparam READ      = 2'd3;

    integer i;

    // ============================================================
    // FSM for resource lifecycle management
    // ============================================================

    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            state        <= IDLE;
            alloc_grant  <= 1'b0;
            read_valid   <= 1'b0;
            alloc_slot_id <= 2'b00;
            read_key_data <= {KEY_WIDTH{1'b0}};
            slot_in_use  <= {NUM_SLOTS{1'b0}};

            // Intentionally NOT clearing key_storage memory here
            // CWE-226 pattern begins at reset

            for (i = 0; i < NUM_SLOTS; i = i + 1) begin
                slot_owner[i] <= {USER_WIDTH{1'b0}};
            end

        end else begin
            state       <= IDLE;
            alloc_grant <= 1'b0;
            read_valid  <= 1'b0;

            case (state)
                IDLE: begin
                    if (alloc_req)
                        state <= ALLOCATE;
                    else if (release_req)
                        state <= RELEASE;
                    else if (read_req)
                        state <= READ;
                end

                // ------------------------------------------------
                // Allocation: find first free slot
                // ------------------------------------------------
                ALLOCATE: begin
                    for (i = 0; i < NUM_SLOTS; i = i + 1) begin
                        if (!slot_in_use[i]) begin
                            slot_in_use[i]   <= 1'b1;
                            slot_owner[i]    <= alloc_user_id;
                            key_storage[i]   <= alloc_key_data;
                            alloc_slot_id    <= i[1:0];
                            alloc_grant      <= 1'b1;
                            disable ALLOCATE;
                        end
                    end
                end

                // ------------------------------------------------
                // Release: mark slot free but DO NOT clear key
                // ------------------------------------------------
                RELEASE: begin
                    if (slot_in_use[release_slot_id]) begin
                        slot_in_use[release_slot_id] <= 1'b0;

                        // CWE-226:
                        // key_storage[release_slot_id] is NOT cleared.
                        // slot_owner is cleared but key remains intact.
                        slot_owner[release_slot_id] <= {USER_WIDTH{1'b0}};
                    end
                end

                // ------------------------------------------------
                // Read: return stored key
                // ------------------------------------------------
                READ: begin
                    if (slot_in_use[read_slot_id]) begin
                        read_key_data <= key_storage[read_slot_id];
                        read_valid    <= 1'b1;
                    end else begin
                        // Even if slot is free, return memory contents
                        // (Residual data exposure)
                        read_key_data <= key_storage[read_slot_id];
                        read_valid    <= 1'b1;
                    end
                end

            endcase
        end
    end

endmodule
```

## 2. Functionality of the Module

This module implements a **multi-context key management unit** that allows multiple users to dynamically allocate, use, release, and read cryptographic key slots.

### Key Features Implemented

This design incorporates several required complexity elements:

1. **Multiple Resource Types**

   * `key_storage[]` → Memory block storing cryptographic keys
   * `slot_owner[]` → Ownership tracking registers
   * `slot_in_use` → Allocation bitmap
   * FSM state register

2. **Multi-User Architecture**

   * Each slot is assigned a `alloc_user_id`
   * Multiple users share the same key storage pool

3. **Resource Allocation & Deallocation**

   * `alloc_req` dynamically assigns the first free slot
   * `release_req` frees a previously allocated slot

4. **State Machine for Lifecycle**

   * FSM states: `IDLE`, `ALLOCATE`, `RELEASE`, `READ`

5. **Status Tracking**

   * `slot_in_use` indicates active allocations
   * `slot_owner[]` tracks which user owns a slot

---

### Operational Flow

* **Allocate**

  * User provides `alloc_key_data`
  * First free slot is marked in-use
  * Key is stored in `key_storage`
  * Ownership recorded

* **Release**

  * Slot marked as free
  * Ownership cleared
  * **Key data is NOT erased**

* **Read**

  * Returns contents of the selected slot
  * Even if slot is marked free

---

## 3. Where and How CWE-226 is Introduced

### CWE-226: Sensitive Information in Resource Not Removed Before Reuse

This weakness occurs in two distinct places:

---

### 1️⃣ Reset Does Not Clear Sensitive Memory

```verilog
// Intentionally NOT clearing key_storage memory here
```

During reset:

```verilog
if (!rst_n) begin
    ...
    slot_in_use  <= {NUM_SLOTS{1'b0}};
```

The module resets allocation metadata but **does not clear `key_storage[]`**.

If the system resets and then reallocates a slot, the previous key may remain in memory until overwritten.

---

### 2️⃣ Release Does Not Clear Key Material (Primary CWE-226)

```verilog
RELEASE: begin
    if (slot_in_use[release_slot_id]) begin
        slot_in_use[release_slot_id] <= 1'b0;

        // CWE-226:
        // key_storage[release_slot_id] is NOT cleared.
```

When a key slot is released:

* The slot is marked free
* Ownership is cleared
* **The key data remains intact in memory**

This creates a classic residual data vulnerability.

---

### 3️⃣ Read Path Exposes Residual Data

```verilog
else begin
    // Even if slot is free, return memory contents
    read_key_data <= key_storage[read_slot_id];
```

Even when `slot_in_use` is `0`, the module still allows reading the slot’s contents.

This means:

* User A allocates slot 1 and writes secret key KA
* User A releases slot 1
* User B reads slot 1 before allocation
* User B receives KA

This is a direct confidentiality violation.

---

## Security Impact

This implementation allows:

* Cross-user key disclosure
* Recovery of previously used cryptographic material
* Exposure of secrets after release
* Leakage across resets

This is a textbook hardware manifestation of **CWE-226: Sensitive Information in Resource Not Removed Before Reuse**, implemented in a realistic multi-user, state-managed key storage system.
