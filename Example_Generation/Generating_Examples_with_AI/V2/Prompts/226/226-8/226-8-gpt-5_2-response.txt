## 1. Complete Verilog Module (with intentional CWE-226 weakness)

```verilog
// Multi-context Key Management Unit
// Intentionally vulnerable to CWE-226: Sensitive Information in Resource Not Removed Before Reuse

module key_management_unit #(
    parameter NUM_SLOTS  = 4,
    parameter KEY_WIDTH  = 128,
    parameter USER_WIDTH = 4
)(
    input                         clk,
    input                         rst_n,

    // Allocation interface
    input                         alloc_req,
    input      [USER_WIDTH-1:0]   alloc_user_id,
    input      [KEY_WIDTH-1:0]    alloc_key_data,
    output reg                    alloc_grant,
    output reg [$clog2(NUM_SLOTS)-1:0] alloc_slot_id,

    // Release interface
    input                         release_req,
    input      [$clog2(NUM_SLOTS)-1:0] release_slot_id,

    // Read interface
    input                         read_req,
    input      [$clog2(NUM_SLOTS)-1:0] read_slot_id,
    output reg [KEY_WIDTH-1:0]    read_key_data,
    output reg                    read_valid,

    // Status
    output reg [NUM_SLOTS-1:0]    slot_allocated
);

    // ------------------------------------------------------------
    // Internal resources
    // ------------------------------------------------------------

    // Key storage (sensitive information)
    reg [KEY_WIDTH-1:0] key_mem [0:NUM_SLOTS-1];

    // Ownership tracking
    reg [USER_WIDTH-1:0] owner_id [0:NUM_SLOTS-1];

    // Simple FSM states
    localparam STATE_IDLE     = 2'd0;
    localparam STATE_ALLOCATE = 2'd1;
    localparam STATE_RELEASE  = 2'd2;

    reg [1:0] state, next_state;

    integer i;

    // ------------------------------------------------------------
    // FSM - Sequential
    // ------------------------------------------------------------
    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            state <= STATE_IDLE;
        end else begin
            state <= next_state;
        end
    end

    // ------------------------------------------------------------
    // FSM - Combinational
    // ------------------------------------------------------------
    always @(*) begin
        next_state = state;
        case (state)
            STATE_IDLE: begin
                if (alloc_req)
                    next_state = STATE_ALLOCATE;
                else if (release_req)
                    next_state = STATE_RELEASE;
            end

            STATE_ALLOCATE: begin
                next_state = STATE_IDLE;
            end

            STATE_RELEASE: begin
                next_state = STATE_IDLE;
            end

            default: next_state = STATE_IDLE;
        endcase
    end

    // ------------------------------------------------------------
    // Allocation, Release, and Read Logic
    // ------------------------------------------------------------
    always @(posedge clk or negedge rst_n) begin
        if (!rst_n) begin
            alloc_grant     <= 1'b0;
            alloc_slot_id   <= {($clog2(NUM_SLOTS)){1'b0}};
            read_key_data   <= {KEY_WIDTH{1'b0}};
            read_valid      <= 1'b0;
            slot_allocated  <= {NUM_SLOTS{1'b0}};

            for (i = 0; i < NUM_SLOTS; i = i + 1) begin
                key_mem[i]  <= {KEY_WIDTH{1'b0}};
                owner_id[i] <= {USER_WIDTH{1'b0}};
            end
        end else begin
            alloc_grant <= 1'b0;
            read_valid  <= 1'b0;

            case (state)

                // ------------------------------------------------
                // Allocate first free slot
                // ------------------------------------------------
                STATE_ALLOCATE: begin
                    for (i = 0; i < NUM_SLOTS; i = i + 1) begin
                        if (!slot_allocated[i] && !alloc_grant) begin
                            slot_allocated[i] <= 1'b1;
                            key_mem[i]        <= alloc_key_data;   // store sensitive key
                            owner_id[i]       <= alloc_user_id;
                            alloc_slot_id     <= i[$clog2(NUM_SLOTS)-1:0];
                            alloc_grant       <= 1'b1;
                        end
                    end
                end

                // ------------------------------------------------
                // Release slot (CWE-226 vulnerability here)
                // ------------------------------------------------
                STATE_RELEASE: begin
                    if (slot_allocated[release_slot_id]) begin
                        slot_allocated[release_slot_id] <= 1'b0;
                        // Vulnerability: key_mem and owner_id are NOT cleared
                        // Sensitive key data remains in key_mem
                    end
                end

            endcase

            // ----------------------------------------------------
            // Read logic (no ownership enforcement)
            // ----------------------------------------------------
            if (read_req) begin
                read_key_data <= key_mem[read_slot_id];
                read_valid    <= 1'b1;
            end
        end
    end

endmodule
```

---

## 2. Module Functionality

This module implements a **multi-context Key Management Unit (KMU)** that allows multiple users to dynamically allocate and release key storage slots.

### Key Features:

* **Parameterized design**

  * `NUM_SLOTS` – number of key storage slots
  * `KEY_WIDTH` – width of stored cryptographic keys
  * `USER_WIDTH` – width of user/context identifier

* **Multiple resource types**

  * `key_mem[]` – memory array storing sensitive cryptographic keys
  * `owner_id[]` – register array tracking slot ownership
  * `slot_allocated` – bitmap tracking allocation state

* **Resource allocation & deallocation logic**

  * `alloc_req` allocates the first available slot
  * `release_req` frees a slot

* **FSM managing lifecycle**

  * `STATE_IDLE`
  * `STATE_ALLOCATE`
  * `STATE_RELEASE`

* **Handshaking signals**

  * `alloc_req` / `alloc_grant`
  * `read_req` / `read_valid`

* **Status tracking**

  * `slot_allocated` indicates which slots are in use

### Operational Flow

1. A user asserts `alloc_req` with `alloc_key_data` and `alloc_user_id`.
2. The FSM finds a free slot and stores the key.
3. The slot can later be released using `release_req`.
4. Any slot can be read using `read_req`.

---

## 3. CWE-226 Weakness Explanation

### CWE-226: Sensitive Information in Resource Not Removed Before Reuse

This vulnerability occurs in the **release logic**:

```verilog
STATE_RELEASE: begin
    if (slot_allocated[release_slot_id]) begin
        slot_allocated[release_slot_id] <= 1'b0;
        // Vulnerability: key_mem and owner_id are NOT cleared
    end
end
```

### What Goes Wrong

When a slot is released:

* `slot_allocated[release_slot_id]` is cleared
* BUT:

  * `key_mem[release_slot_id]` still contains the previous cryptographic key
  * `owner_id[release_slot_id]` still contains the previous user ID

The slot is now marked **free**, but its sensitive contents remain intact.

---

### How the Vulnerability Manifests

1. **User A** allocates a slot and stores a secret key.
2. User A releases the slot.
3. The slot becomes available for reuse.
4. **User B** allocates the same slot.
5. Before writing a new key (or by directly reading it), User B can retrieve residual key data.

Additionally:

* The read logic does not enforce ownership validation.
* Any `read_req` can access any slot’s key contents.

---

### Security Impact

* Confidential cryptographic keys remain in hardware after release.
* Cross-user key leakage is possible.
* Violates isolation between security domains.
* Direct example of:

  > "Failure to remove sensitive information before resources are reused."

---

This design intentionally demonstrates a **realistic, industry-pattern CWE-226 hardware weakness** in a shared key management system.
