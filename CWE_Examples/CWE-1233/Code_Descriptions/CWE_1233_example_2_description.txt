This code example is from the HMAC wrapper module from the HACK@DAC'21 OpenPiton SoC 
repository. This bug was inserted as bug #75 for the event. This bug was fixed, but I 
have modified the code to be vulnerable again. I have modified one assignment by switching 
wdata to be assigned to data[0] when the register lock is set to 1 and when the register lock 
is set to 0 the assignment is rejected. This means that unauthorized writes could be possible 
when the register should be locked This weakness could allow an attacker to bypass authentication or create 
.situations where legitimate HMACs will be rejected
